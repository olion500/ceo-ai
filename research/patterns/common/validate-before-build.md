# Validate Before Build

**Category:** common
**Success Rate:** High (found in 95%+ of successful stories)
**Time Investment:** 1-2 weeks
**Difficulty:** Easy for solo founders

## What Is This Pattern?

성공한 거의 모든 사업은 본격적인 개발에 들어가기 전에 아이디어를 검증했습니다. 시간과 돈을 투자하기 전에 실제 수요가 있는지 확인하는 필수 단계입니다.

## How It Works

1. **Step 1: Problem Validation** - 문제가 실제로 존재하고 충분히 고통스러운지 확인
2. **Step 2: Solution Interest** - 당신의 솔루션에 관심이 있는지 테스트
3. **Step 3: Willingness to Pay** - 돈을 낼 의향이 있는지 검증
4. **Step 4: Build Decision** - 충분한 검증 신호가 있으면 빌드 시작

## Why This Works

**검증 없이 빌드하면:**
- 아무도 원하지 않는 것을 만들 위험
- 시간과 돈 낭비
- 감정적 투자로 인한 sunk cost fallacy

**검증하고 빌드하면:**
- 실제 수요 확인
- 런칭 전 이미 관심 있는 고객 확보
- 빌드할 기능 우선순위 명확
- 실패 시 빠른 피봇 가능

## Prerequisites

- **Skills Required:**
  - 고객 인터뷰 능력
  - 랜딩 페이지 제작 (노코드 가능)
  - 데이터 해석 능력
- **Time Required:** 1-2주 (최소)
- **Capital Required:** $0-100 (랜딩 페이지 호스팅 등)
- **Other Requirements:** 타겟 고객에게 접근할 방법

## Common Mistakes

1. **Mistake: "친구와 가족에게만 물어봄"**
   - **Why It Fails:** 편향된 긍정적 피드백
   - **How to Avoid:** 실제 타겟 고객 최소 20명 이상과 대화

2. **Mistake: "좋은 아이디어라고 말하면 검증됐다고 생각"**
   - **Why It Fails:** 말과 행동은 다름 (Lip service)
   - **How to Avoid:** 돈을 내거나, 이메일을 주거나, 대기자 명단에 등록하는 등 실제 행동 요구

3. **Mistake: "완벽한 검증까지 기다림"**
   - **Why It Fails:** 100% 확신은 불가능, 분석 마비
   - **How to Avoid:** "충분한 신호" 기준 정하기 (예: 50명 이메일 가입)

## When to Use This Pattern

✅ **Good Fit When:**
- 새로운 사업 아이디어가 있을 때
- B2B 제품 (고객이 명확할 때)
- 비용/시간이 많이 드는 빌드일 때
- 시장이 불확실할 때

❌ **Bad Fit When:**
- 이미 명확한 수요가 있는 시장 (예: 기존 제품의 명백한 개선)
- 검증 자체가 빌드보다 더 어려울 때 (매우 드묾)
- 타이밍이 중요해서 빠른 런칭이 필수일 때
