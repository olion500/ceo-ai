# Cloud to Self-Hosting Success Patterns

## Executive Summary

AWS 클라우드 비용을 직접 호스팅으로 전환하여 수익을 개선한 사례는 **2015년 이후 뚜렷한 트렌드**로 자리잡았습니다. 초기에는 Dropbox 같은 대기업만 가능했지만, 2024년 현재는 **중소 규모 SaaS 회사도 충분히 실행 가능**합니다.

**핵심 인사이트:**
- 클라우드 비용이 연간 $200K 이상이면 셀프 호스팅 고려 시작
- 데이터 집약적 비즈니스는 클라우드 경제성이 근본적으로 맞지 않음
- 2015년 vs 2024년: 하드웨어 가격 하락 + 오픈소스 성숙 = 진입장벽 급감
- 100% 전환이 아닌 **80-90% 하이브리드**가 최적

## 분석 대상 주요 사례

### 1. 37signals (Basecamp, HEY) - 2024

**절감액:** $10M+ (5년간)
**규모:** ~80명, ~$100M ARR
**전략:** AWS → Dell 서버 (기존 데이터센터 랙 활용)

**특징:**
- 가장 최근 사례 (2023-2024)
- 중소 규모도 가능함을 증명
- DHH의 적극적인 공개 전략
- 추가 인력 없이 동일 팀으로 운영

### 2. Ahrefs - 2023

**절감액:** $400M+ (3년간, vs. 클라우드 대안)
**규모:** SEO 도구, $257M revenue (3년)
**전략:** 처음부터 셀프 호스팅 (클라우드 미사용)

**특징:**
- 가장 큰 절감액
- 처음부터 경제성 계산 후 결정
- 850+ 서버, 200 PB 저장소
- 싱가포르 코로케이션
- **클라우드 사용시 비즈니스 자체가 불가능했을 것**

### 3. Dropbox - 2018

**절감액:** $74.6M (2년간)
**규모:** 500M users, 500 PB at migration
**전략:** AWS S3 → 자체 "Magic Pocket" 인프라

**특징:**
- 최초의 대규모 공개 사례 (2015-2016)
- 커스텀 인프라 구축 (Magic Pocket)
- IPO 전 마진 개선 전략
- 업계 인식 변화의 시작점

## 공통 패턴 분석

### 패턴 1: 경제성 임계점 (Economic Tipping Point)

**발견:**
모든 사례에서 특정 규모 이상에서 클라우드 경제성이 역전됩니다.

**임계점 기준:**

| 비즈니스 규모 | 연간 클라우드 비용 | 권장 행동 |
|--------------|-----------------|---------|
| 소규모 | < $100K | 클라우드 유지 |
| 중소규모 | $100K - $500K | 경제성 계산 시작 |
| 중규모 | $500K - $5M | 진지하게 마이그레이션 고려 |
| 대규모 | > $5M | 대부분 마이그레이션이 유리 |
| 초대규모 | > $50M | 거의 확실히 마이그레이션 필요 |

**37signals 예시:**
```
클라우드: $3.2M/년
하드웨어: $700K (1회) + $1.3M/년 (운영)
손익분기: 4개월
5년 절감: $10M+
```

**공식:**
```python
def should_migrate(annual_cloud_cost, scale_tb):
    # Rule of thumb
    if annual_cloud_cost > 200_000:
        hardware_cost = annual_cloud_cost * 0.3  # 대략 30%
        breakeven_months = (hardware_cost / (annual_cloud_cost / 12)) * 12

        if breakeven_months < 24:  # 2년 이내 손익분기
            return "진지하게 고려"

    if scale_tb > 100 and annual_cloud_cost > 1_000_000:
        return "강력히 권장"

    return "클라우드 유지"
```

### 패턴 2: 데이터 집약도가 핵심 (Data Intensity Matters Most)

**발견:**
컴퓨트 중심 워크로드보다 **스토리지/대역폭 집약적** 워크로드가 셀프 호스팅에 유리합니다.

**클라우드 비용 구조:**

**Ahrefs 분석 (서버당 월비용):**
| 항목 | AWS 비용 | 비중 | 셀프호스팅 | 비중 |
|------|---------|------|----------|------|
| 스토리지 (240TB) | $11,486 | 65% | $1,025 (HW 상각 포함) | 66% |
| 컴퓨트 | $5,607 | 32% | - | - |
| 대역폭 | $464 | 3% | $524 (전체 랙/ISP) | 34% |
| **합계** | **$17,557** | **100%** | **$1,550** | **100%** |

**인사이트:**
- AWS는 스토리지에 65% 비용 (선형 스케일)
- 셀프호스팅은 HW 상각 (고정비용화)
- **스토리지 많을수록 셀프호스팅 유리**

**비즈니스 타입별 적합도:**

| 비즈니스 타입 | 데이터 집약도 | 셀프호스팅 적합도 | 사례 |
|--------------|-------------|----------------|------|
| 파일 저장 | 매우 높음 | ⭐⭐⭐⭐⭐ | Dropbox, Backblaze |
| 웹 크롤러 | 매우 높음 | ⭐⭐⭐⭐⭐ | Ahrefs |
| 비디오 플랫폼 | 높음 | ⭐⭐⭐⭐ | YouTube (self-hosted) |
| SaaS (DB 중심) | 중간 | ⭐⭐⭐ | 37signals |
| 분석/ML | 중간 | ⭐⭐⭐ | - |
| 서버리스 앱 | 낮음 | ⭐ | - |
| API 서비스 | 낮음 | ⭐⭐ | - |

### 패턴 3: 하이브리드가 최적 (Hybrid is Optimal)

**발견:**
100% 클라우드 탈출이 아닌 **80-90% 마이그레이션**이 가장 효율적입니다.

**각 사례의 하이브리드 전략:**

**37signals:**
- ✅ 셀프호스팅: 코어 앱, 데이터베이스, 스토리지 (대부분)
- ☁️ 클라우드: CDN, DNS, edge locations
- 비율: ~80% 셀프, 20% 클라우드

**Ahrefs:**
- ✅ 셀프호스팅: 크롤러, 데이터 스토리지, 처리 (95%+)
- ☁️ 클라우드: 프론트엔드 글로벌 배포
- 비율: ~95% 셀프, 5% 클라우드

**Dropbox:**
- ✅ 셀프호스팅: 미국 사용자 스토리지 (90%)
- ☁️ 클라우드: 유럽 리전, CDN
- 비율: ~90% 셀프, 10% 클라우드

**최적 분할 기준:**

```
셀프호스팅에 적합:
- 예측 가능한 워크로드 (24/7 일정)
- 대용량 스토리지
- 높은 대역폭
- 지역 집중 (미국/아시아 등)

클라우드 유지:
- 글로벌 배포 (CDN, edge)
- 급격한 스파이크 대응
- 규제/컴플라이언스
- 실험적 워크로드
```

### 패턴 4: 점진적 마이그레이션 (Gradual Migration)

**발견:**
Big Bang 방식이 아닌 **단계적 마이그레이션**이 성공의 핵심입니다.

**37signals 타임라인:**
```
2022 Q3: 경제성 분석 및 결정
2022 Q4: 하드웨어 구매 및 설치
2023 Q1-Q2: 앱 마이그레이션 (7개 앱)
2023 Q3: 마이그레이션 완료
2023 Q4: 장기 계약 만료
2024: 첫 번째 완전한 절감 연도
2025 예정: S3 스토리지 마이그레이션
```

**Dropbox 타임라인:**
```
2015 초: Magic Pocket 설계 시작
2015 중: 3개 데이터센터 구축
2015 하: 데이터 마이그레이션 시작 (1% → 5% → ...)
2016 Q4: 90% 마이그레이션 완료
2017-2018: 최적화 및 비용 절감 실현
```

**권장 단계:**

**Phase 0: 분석 (1-2개월)**
- 현재 클라우드 비용 상세 분석
- 워크로드 특성 파악
- 셀프호스팅 비용 견적
- ROI 계산

**Phase 1: 컴퓨트 마이그레이션 (3-6개월)**
- 스테이트리스 앱부터 시작
- 클라우드 스토리지는 유지 (안전망)
- 운영 경험 축적
- 팀 역량 검증

**Phase 2: 검증 (3-6개월)**
- 모니터링 및 최적화
- 비용 절감 실측
- 추가 워크로드 이관 준비

**Phase 3: 스토리지 마이그레이션 (6-12개월)**
- 가장 큰 비용 절감 구간
- 리스크도 가장 높음
- 점진적 데이터 이동
- 충분한 검증 후 완전 전환

**Phase 4: 최적화 (지속)**
- 10-20%는 클라우드 유지 (하이브리드)
- 지속적 비용 최적화
- 성능 튜닝

### 패턴 5: 팀 규모는 불변 (No Additional Headcount)

**발견:**
"셀프호스팅 = 많은 인력 필요"는 **클라우드 벤더의 FUD**입니다.

**37signals:**
- 마이그레이션 전후 동일 팀
- 추가 채용 없음
- DHH: "팀은 그대로, 관리 업무량도 비슷"

**운영 시간 투자 (PostgreSQL 셀프호스팅 예시):**
```
초기 설정: ~40시간 (1주)
월간 유지보수: ~30분 (HA 시스템)
주간 태스크: ~10분 (백업 검증, 슬로우 쿼리)

연간 총 시간: 40 + (30분 × 12) + (10분 × 52) ≈ 55시간
= 1명이 연간 약 1주일
```

**왜 이게 가능한가?**

1. **클라우드 관리도 시간 걸림:**
   - 비용 최적화
   - 서비스 설정 및 모니터링
   - 장애 대응
   - **시간 투자는 비슷함**

2. **도구의 성숙:**
   - Kubernetes, Docker
   - Prometheus, Grafana
   - Terraform, Ansible
   - **오픈소스 도구로 클라우드 수준 자동화**

3. **예측 가능한 워크로드:**
   - 급격한 스케일링 불필요
   - 용량 계획 가능
   - **클라우드의 탄력성 대부분 불필요**

### 패턴 6: 하드웨어 표준화 (Hardware Standardization)

**발견:**
수백 가지 인스턴스 타입이 아닌 **1-2가지 표준 서버 스펙**이 더 효율적입니다.

**Ahrefs 접근:**
- 850+ 서버 거의 동일 스펙
- 2TB RAM, 16 × 15TB NVMe, Dual 100G NIC
- **대량 구매 할인 + 운영 단순화**

**37signals 접근:**
- Dell 서버 표준 스펙
- 범용 하드웨어 (특수 장비 아님)
- **교체/확장 용이**

**클라우드 vs 셀프호스팅 철학:**

| 항목 | 클라우드 | 셀프호스팅 |
|------|---------|----------|
| 인스턴스 종류 | 수백 가지 | 1-2 가지 |
| 최적화 방식 | 워크로드별 인스턴스 선택 | 범용 서버에 소프트웨어 최적화 |
| 확장 방식 | 인스턴스 타입 변경 | 서버 추가 |
| 복잡도 | 높음 (선택의 폭) | 낮음 (표준화) |

**예시:**
```
# 클라우드 방식:
- 웹서버: c5.xlarge
- DB: r5.2xlarge
- 캐시: r5.large
- 워커: m5.large
→ 4가지 인스턴스 관리

# 셀프호스팅 방식:
- 모든 워크로드: 표준 서버 1종
→ 1가지만 관리, 소프트웨어로 최적화
```

### 패턴 7: 코로케이션 활용 (Colocation Strategy)

**발견:**
자체 데이터센터 구축이 아닌 **코로케이션**이 최적 선택입니다.

**코로케이션 장점:**

| 항목 | 자체 DC | 코로케이션 | 클라우드 |
|------|---------|-----------|---------|
| 초기 투자 | 매우 높음 | 중간 | 낮음 |
| 전력/냉각 | 직접 관리 | 제공됨 | 제공됨 |
| 네트워크 | 직접 구축 | Tier-1 제공 | 제공됨 |
| 물리 보안 | 직접 관리 | 제공됨 | 제공됨 |
| 확장성 | 제한적 | 랙 추가 | 무제한 |
| 월 비용 | 고정비 높음 | 랙당 요금 | 사용량 기반 |

**Ahrefs: 싱가포르 코로케이션**
- 서버당 월 $524 (DC + ISP + 네트워크)
- 전력, 냉각, 보안 모두 포함
- Tier-1 네트워크 연결

**37signals: 기존 DC 랙 활용**
- 이미 보유한 데이터센터 공간 사용
- 추가 랙/전력 비용 없음
- **절감액이 예상 초과한 이유**

**코로케이션 선택 기준:**
```python
def choose_infrastructure(annual_cost, scale):
    if annual_cost < 200_000:
        return "클라우드 유지"
    elif annual_cost < 1_000_000:
        return "코로케이션 고려"
    elif annual_cost < 10_000_000:
        return "코로케이션 강력 추천"
    else:
        return "코로케이션 or 자체 DC"
```

## 비용 절감 메커니즘

### 1. 스토리지 비용 구조의 차이

**AWS S3 비용 (2024):**
```
표준 스토리지: $0.023/GB/월
= $23/TB/월
= $23,000/PB/월

10 PB 저장:
= $230,000/월
= $2,760,000/년
```

**셀프호스팅 (Dropbox 예시):**
```
하드웨어 투자: $53M (500 PB)
= $106,000/PB
= $106/TB (5년 상각)

10 PB 저장 (5년):
초기: $1,060,000
연간 운영: ~$200,000
5년 총: $2,060,000

vs AWS 5년: $13,800,000
절감: $11,740,000 (85% 절감)
```

### 2. 컴퓨트 비용 비교

**AWS RDS vs 셀프호스팅 (PostgreSQL):**

| 스펙 | AWS RDS | 자체 서버 (Hetzner) |
|------|---------|---------------------|
| 4 vCPU, 32GB RAM | $328/월 | - |
| 32 core, 256GB RAM | ~$2,500/월 | $220/월 |
| 비율 | - | **11배 저렴** |

### 3. 대역폭 비용

**AWS 송신(Egress) 비용:**
```
처음 10 TB: $0.09/GB
= $90/TB
= $900/월 (10TB)

Ahrefs 같은 크롤러: 수십-수백 TB/월
= $수만-수십만/월
```

**코로케이션:**
```
랙당 요금에 포함 (unlimited)
or 매우 저렴한 추가 요금
```

### 4. 하드웨어 가격 하락

**2015 (Dropbox 시대):**
- 커스텀 인프라 필요
- 고가의 전문 장비
- Magic Pocket 개발 필요

**2024 (37signals 시대):**
- 범용 Dell 서버
- NVMe 드라이브 저렴화
- 오픈소스 소프트웨어 성숙

**예시: 1TB NVMe SSD 가격**
```
2015: ~$400
2020: ~$130
2024: ~$80

→ 5배 저렴해짐
```

## 시기별 전략 차이

### 2015-2018: 대기업 전용 (Dropbox Era)

**특징:**
- 거대한 초기 투자 필요
- 커스텀 인프라 개발
- 수백 PB 스케일
- **중소기업은 불가능**

**문턱:**
- 투자: $50M+
- 규모: 100+ PB
- 팀: 전문 인프라 엔지니어 수십명

### 2019-2022: 과도기

**특징:**
- 하드웨어 가격 하락
- 오픈소스 도구 성숙
- 클라우드 가격 상승

## 실전 의사결정 프레임워크

### Step 1: 자격 요건 체크

```python
def qualified_for_migration():
    checklist = {
        "클라우드 비용 > $200K/년": False,
        "예측 가능한 워크로드": False,
        "데이터 집약적": False,
        "성숙한 제품": False,
        "인프라 역량": False,
    }

    # 최소 3개 이상 True 필요
    if sum(checklist.values()) >= 3:
        return "마이그레이션 고려 가능"
    else:
        return "클라우드 유지 권장"
```

### Step 2: 경제성 계산

**간단한 ROI 계산:**
```python
# 입력값
annual_cloud_cost = 3_200_000  # 37signals 예시
storage_tb = 10_000  # 10 PB

# 셀프호스팅 비용 추정
hardware_cost = annual_cloud_cost * 0.25  # 대략 25%
annual_operation = annual_cloud_cost * 0.40  # 대략 40%

# ROI
total_5yr_cloud = annual_cloud_cost * 5
total_5yr_self = hardware_cost + (annual_operation * 5)
savings_5yr = total_5yr_cloud - total_5yr_self
breakeven_months = hardware_cost / (annual_cloud_cost / 12)

print(f"5년 절감: ${savings_5yr:,.0f}")
print(f"손익분기: {breakeven_months:.1f}개월")

# 37signals 실제:
# 5년 절감: $10,000,000
# 손익분기: 4개월
```

### Step 3: 리스크 평가

**높은 리스크 (마이그레이션 보류):**
- ❌ 초기 스타트업 (PMF 미달성)

- ❌ 인프라 경험 전무

**중간 리스크 (신중하게 진행):**

- ⚠️ 글로벌 사용자 (낮은 지연시간 필요)
- ⚠️ 복잡한 아키텍처 (수십개 서비스)

**낮은 리스크 (적극 추진):**

## 흔한 반론과 대응

### 반론 1: "기회비용이 크지 않나요?"

**반론:**
엔지니어가 인프라 관리할 시간에 기능 개발해야 하지 않나?

**대응 (37signals):**
- 클라우드도 관리 시간 필요
- 비용 최적화, 설정, 모니터링
- **시간 투자는 비슷함**
- 연 $2M 절감 >> 엔지니어 몇 명 비용

**수치:**
```
클라우드 관리: 주 5시간 (비용 최적화, 장애 대응)
셀프호스팅 관리: 주 6시간 (모니터링, 유지보수)
차이: 주 1시간

연 절감: $2,000,000
엔지니어 1명 비용: ~$150,000
ROI: 13배
```

### 반론 2: "스케일링이 어렵지 않나요?"

**반론:**
클라우드는 무제한 확장 가능한데, 서버는 한계가 있지 않나?

**대응 (37signals, Ahrefs):**
- 대부분 회사는 무제한 스케일 불필요
- 성장은 예측 가능 (월별 추세)
- 필요시 서버 추가 구매 (여전히 저렴)
- Ahrefs: 850 서버로 전세계 크롤링

**현실:**
```
"무제한 스케일" 필요한 경우:
- 바이럴 앱 (급격한 사용자 증가)
- 이벤트 기반 (월드컵, 대선 등)
- 실제 비율: < 1% 회사

"예측 가능한 성장":
- 대부분의 SaaS (월 5-20% 성장)
- 실제 비율: > 95% 회사
```

### 반론 3: "지리적 중복성은 어떻게?"

**반론:**
클라우드는 자동으로 글로벌 분산이 되는데?

**대응 (Dropbox, 37signals):**
- 3개 데이터센터로 충분 (미국 동/서/중부)
- 대부분 사용자는 한 지역 집중
- 진짜 글로벌 분산이 필요한 경우는 드묾
- 필요시 클라우드 CDN 하이브리드

**Dropbox 예시:**
- 3개 DC (CA, VA, TX)
- 미국 사용자 99.9% 커버
- 유럽은 여전히 AWS 사용 (하이브리드)

### 반론 4: "우리 회사는 너무 작아요"

**반론:**
Dropbox, 37signals 같은 큰 회사나 가능한 거 아닌가?

**대응:**
- 37signals: 80명 규모
- 클라우드 $200K/년이면 고려 가능
- Hetzner 전용 서버: $40/월부터
- **중소기업이 더 유리할 수도 있음**

**중소기업 이점:**
```
대기업:
- 복잡한 아키텍처
- 많은 레거시
- 의사결정 느림

중소기업:
- 단순한 아키텍처
- 빠른 결정
- 마이그레이션 쉬움
→ 상대적으로 더 쉬울 수 있음
```

### 반론 5: "나중에 클라우드로 돌아가면?"

**반론:**
셀프호스팅 했다가 다시 클라우드로 가면 손해 아닌가?

**대응:**
- 하드웨어는 재판매 가능
- 코로케이션은 월 계약 (유연함)
- **클라우드 → 셀프 → 클라우드 모두 가능**
- 37signals: 5년 쓰고 재결정해도 $10M 절감

**리스크 관리:**
```
하드웨어 상각: 5년
→ 2년 쓰고 클라우드 복귀해도 40% 절감
→ 3년 쓰면 손익분기
→ 5년 쓰면 최대 절감
```

## 메타 인사이트

### 1. 패러다임의 변화

**2010년대 통념:**
> "모든 회사는 클라우드로 가야 한다"

**2020년대 현실:**
> "회사마다 최적 인프라는 다르다"

**변화의 원인:**
- 클라우드 가격 상승
- 하드웨어 가격 하락
- 오픈소스 성숙
- **경제성 역전**

### 2. "Cloud Repatriation" 트렌드

**통계:**
- 2024 Barclays CIO Survey: 86% CIO가 일부 워크로드 클라우드 → 온프렘 이동 계획
- 역대 최고 수치
- **명확한 트렌드**

**이동하는 워크로드:**
1. 대용량 스토리지
2. 데이터베이스
3. 컴퓨트 집약적 배치 작업
4. 예측 가능한 서비스

**클라우드에 남는 것:**
1. CDN / Edge
2. 서버리스 함수
3. 실험적 프로젝트
4. 스파이크 워크로드

### 3. 벤더 락인 vs 자유

**Ahrefs 관점:**
> "클라우드는 편리하지만 벤더 락인을 가져온다"

**락인의 형태:**
- 독점 API (AWS-specific)
- 가격 변경 (일방적)
- 서비스 종료 (강제 마이그레이션)
- **협상력 상실**

**셀프호스팅의 자유:**
- 코로케이션 업체 교체 가능
- 하드웨어 업그레이드 자유
- 가격 협상 가능
- **통제권 유지**

### 4. 기술부채의 역설

**일반적 생각:**
> 셀프호스팅 = 기술부채 증가

**실제 경험:**
> 클라우드도 기술부채 발생

**클라우드 기술부채:**
- AWS-specific 코드
- 복잡한 IAM 설정
- 비용 최적화 노하우
- **마이그레이션 시 모두 버려짐**

**셀프호스팅:**
- 표준 기술 (Linux, PostgreSQL, etc.)
- 이식 가능한 지식
- **다음 회사에서도 쓸 수 있는 스킬**

### 5. 리크루팅 효과 (숨은 이점)

**37signals 발견:**
엔지니어들이 "진짜 인프라" 다루는 회사에 관심

**이유:**
- 클라우드만 경험 = 상품화된 스킬
- 셀프호스팅 = 깊은 이해 필요
- **학습 기회로 인식**

**차별화:**
```
채용 포스팅:
"We run our own infrastructure"
→ 시스템 엔지니어 지원 증가
→ 차별화된 브랜드
```

### 마이그레이션 고려 시작 체크리스트

```
□ 연간 클라우드 비용 > $200,000
□ 월별 클라우드 비용이 예측 가능 (±20% 이내)
□ 데이터 스토리지 > 10 TB
□ 제품이 성숙 단계 (PMF 달성)
□ 인프라 관리 경험 보유 (사내 또는 컨설턴트)
□ 사용자가 특정 지역에 집중 (글로벌 필수 아님)
□ 급격한 스파이크 없음 (성장률 월 <50%)

5개 이상 체크 → 진지하게 고려
7개 체크 → 강력 추천
```

### ROI 계산 체크리스트

```
□ 현재 클라우드 비용 상세 분석 완료
□ 워크로드별 비용 분해 완료 (컴퓨트/스토리지/대역폭)
□ 셀프호스팅 하드웨어 견적 3곳 이상 받음
□ 코로케이션 비용 견적 확보
□ 5년 TCO 비교 완료
□ 손익분기점 < 24개월 확인
□ 리스크 분석 완료

모두 체크 → 경영진 보고 및 승인 요청
```

### 실행 준비 체크리스트

```
□ 예산 확보 (하드웨어 + 코로케이션 초기비)
□ 마이그레이션 팀 구성 (2-4명)
□ 단계적 마이그레이션 계획 수립
□ 롤백 계획 수립
□ 모니터링 도구 선정
□ 코로케이션 업체 선정 및 계약
□ 하드웨어 발주

모두 체크 → 마이그레이션 시작
```

## 핵심 Takeaways

### 1. 경제성은 명확하다
- $200K/년 이상: 셀프호스팅 검토 시작
- $1M/년 이상: 대부분 마이그레이션이 유리
- **ROI는 수개월 ~ 2년 이내**

### 2. 기술적 난이도는 낮아졌다
- 2015 Dropbox: 커스텀 인프라 필요
- 2024 현재: 범용 하드웨어 + 표준 소프트웨어
- **중소기업도 충분히 가능**

### 3. 팀 규모는 문제가 아니다
- 추가 인력 불필요
- 클라우드 관리 vs 셀프호스팅 관리 시간 비슷
- **"인력 부족"은 핑계**

### 4. 하이브리드가 정답이다
- 100% 전환 불필요
- 80-90% 마이그레이션으로 대부분 절감
- **클라우드는 진짜 강점 영역에만 사용**

### 5. 데이터 집약도가 핵심이다
- 스토리지/대역폭 높을수록 유리
- 컴퓨트 중심은 클라우드가 나을 수도
- **비즈니스 모델 분석이 우선**

## 결론

AWS 클라우드에서 셀프 호스팅으로의 전환은 **2024년 현재 명확한 트렌드**이며, 더 이상 대기업만의 전유물이 아닙니다.

**핵심 메시지:**
1. **경제성은 명확:** 연 $200K 이상이면 고려, $1M 이상이면 강력 권장
2. **기술적 실현 가능성:** 범용 하드웨어 + 오픈소스로 중소기업도 가능
3. **리스크는 관리 가능:** 단계적 마이그레이션 + 하이브리드 전략
4. **ROI는 빠름:** 대부분 1-2년 이내 손익분기

**다음 세대 인프라 전략:**
- 클라우드로 시작 (빠른 실험, 낮은 초기비용)
- 성장하며 비용 모니터링
- $200K/년 도달 시 셀프호스팅 검토
- 단계적으로 80-90% 마이그레이션
- **하이브리드 모델이 최종 목표**
